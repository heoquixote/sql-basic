## 학습 목표

- SQL 쿼리가 작성되는 순서와 실행되는 순서를 구분할 수 있습니다.
- 각 절이 어떤 데이터를 대상으로 동작하는지 이해할 수 있습니다.

## 목차

1. 작성순서
2. 작동순서
3. 왜 이 차이가 중요한가

## 1. 작성순서

- **`SELECT -> FROM -> WHERE -> GROUP BY -> HAVING -> ORDER BY`**

우리가 SQL을 직접 작성할 때는 보통 위 순서로 적습니다.

- `SELECT`: 어떤 컬럼을 보여줄지 결정
- `FROM`: 어떤 테이블에서 가져올지 결정
- `WHERE`: 어떤 행만 남길지 조건 지정
- `GROUP BY`: 어떤 기준으로 묶을지 결정
- `HAVING`: 묶인 결과에 조건 지정
- `ORDER BY`: 최종 결과를 어떤 순서로 정렬할지 결정

즉, 작성순서는 사람이 읽고 쓰기 편한 순서라고 이해하면 됩니다.

## 2. 작동순서

- **`FROM -> ON -> JOIN -> WHERE -> GROUP BY -> HAVING -> SELECT -> DISTINCT -> ORDER BY`**

하지만 DB가 실제로 데이터를 처리하는 흐름은 위와 다릅니다.

- `FROM`: 먼저 어떤 테이블을 기준으로 볼지 정합니다.
- `ON`: 조인 조건을 확인합니다.
- `JOIN`: 테이블을 실제로 결합합니다.
- `WHERE`: 결합된 결과에서 행 단위 필터링을 합니다.
- `GROUP BY`: 남은 행들을 그룹으로 묶습니다.
- `HAVING`: 그룹화된 결과를 다시 필터링합니다.
- `SELECT`: 그 다음에야 최종 출력 컬럼을 만듭니다.
- `DISTINCT`: 출력 결과에서 중복을 제거합니다.
- `ORDER BY`: 마지막에 정렬합니다.

즉, SQL은 `먼저 데이터를 모으고 -> 걸러내고 -> 묶고 -> 보여주는` 흐름으로 작동합니다.

## 3. 왜 이 차이가 중요한가

이 순서를 이해하지 못하면 다음처럼 자주 헷갈립니다.

- 왜 `WHERE`에서는 집계 함수를 바로 못 쓰는가
- 왜 `HAVING`이 따로 필요한가
- 왜 `SELECT`에서 만든 별칭을 앞 단계에서 못 쓰는가

예를 들어:

- `WHERE`는 `GROUP BY`보다 먼저 실행되므로 집계 결과를 바로 조건으로 걸 수 없습니다.
- 그래서 집계 이후 조건은 `HAVING`을 사용해야 합니다.
- 또 `SELECT` 별칭은 거의 마지막 단계에서 만들어지므로, 앞 단계인 `WHERE`에서는 아직 존재하지 않는 값입니다.

실무에서는 SQL 문장을 눈으로만 읽지 않고,
`지금 DB가 어느 단계까지 처리했는가`를 함께 떠올릴 수 있어야 쿼리 오류를 훨씬 덜 내게 됩니다.
